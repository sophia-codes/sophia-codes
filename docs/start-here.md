# start here

> one problem. one lane. one clear beginning.

if you want to work with sophia, the best first step is simple.

do not bring everything at once.
bring one real problem.

---

## a strong starting point looks like this

choose one of the following:

- a repo that needs clarity
- a feature that needs a clean execution plan
- an architecture that needs review
- a phase or roadmap that has drifted from reality
- a codebase area that needs cleanup and stronger boundaries
- an ai product idea that needs a real implementation shape

---

## what to provide

bring:
- the current objective
- the relevant repo, issue, pr, or doc
- the key constraints
- the desired outcome

that is enough to start producing useful work.

---

## good first engagements

strong first engagements are usually:
- bounded
- reviewable
- high-signal
- attached to real work already in motion

examples:
- audit this repo area and recommend the next best move
- refine this architecture before implementation expands
- break this initiative into execution order
- review this pr for clarity and maintainability
- rewrite this technical doc so it can actually guide the team

---

## what happens next

once the first engagement is clear, sophia can help with:
- follow-up planning
- deeper implementation support
- continued review and refinement
- documentation and execution stewardship across the work

common artifact shapes include:
- architecture reviews
- execution plans
- repo audits
- pr reviews
- decision records
- issue or documentation drafts

start narrow.
expand where the signal is strong.

---

## closing line

> the best beginning is not the biggest one. it is the clearest one.
